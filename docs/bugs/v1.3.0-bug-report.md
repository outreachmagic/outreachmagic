## Bug #1: Manifest includes root-level files → update 404

**Severity:** BLOCKER

`update-manifest.json` lists `SECURITY.md` and `references/command-reference.md` in `files`, but they're root-level files, not in `scripts/`. `update_download_names()` tries to download `{scripts_base}/SECURITY.md` → 404.

**Fix applied (`06ea8e2`):** Switched `update_download_names()` from blacklist (`ROOT_SKILL_FILES`) to whitelist (`UPDATE_MANIFEST_FILES`). Only `.py` files and `VERSION` are downloaded.

## Bug #8: `UPDATE_SCRIPT_FILES` whitelist prevents new `.py` modules from installing

**Severity:** Medium — blocks adding new Python modules via `pipeline.py update`

**Root cause:**
```python
UPDATE_SCRIPT_FILES = tuple(
    sorted(p.name for p in Path(__file__).resolve().parent.glob("*.py"))
)
```

This reads `.py` files from the **locally installed** scripts directory. When `update_download_names()` filters the remote manifest by this list, any new `.py` file (like `reprocess.py`) is excluded because it doesn't exist locally yet. This is a bootstrapping/catch-22 problem.

**Evidence:** After `pipeline.py update --channel main`, the remote manifest had 41 files including `reprocess.py`, but only 34 `.py` files were installed — `reprocess.py` was skipped.

**Fix:**
```python
# Replace the whitelist filter with extension-based matching:
return sorted(
    name for name in manifest_files
    if name.endswith('.py') or name == 'VERSION'
)
```

This ensures any `.py` file in the manifest gets installed regardless of whether it existed in the previous version.

## Bug #12: Prosp campaign_ids are UUIDs, not resolved to campaign names

**Severity:** Medium — can't attribute LinkedIn outreach to named campaigns in stats

Three Prosp campaign UUIDs across 1,541 events. These are Prosp's internal sequence IDs.

**Root cause:** The D1 relay enriches the event payload with `campaign_id` (UUID) during replay, but does NOT include a `campaign` or `campaign_name` string. The extracted metadata for Prosp events has `campaign_id` set but `campaign` is `None`:

```
Prosp event metadata keys: ['body', 'campaign_id', 'platform', 'relay_id', ...]
                              ✅ UUID present        ❌ No campaign name
```

The D1 relay has the ability to map UUID → campaign name (it must, to add the UUID in the first place), but it's not passing the name through to the payload. The fix needs to happen in the D1 relay's replay endpoint — when it adds `campaign_id`, it should also add `campaign_name` to the payload. Then the local extractor (`_PROSP_SPEC` already maps `eventData.campaignName`, `campaignName`, `eventData.campaign`, `campaign`) would pick it up and store it as `$.campaign` in metadata_json.

**Events affected:**
| UUID | Events | Likely Campaign |
|---|---|---|
| `43795293-6fcf-444b-b246-04671a947fcd` | 1,284 | One of the 4 NACE26 campaigns |
| `71381e3d-2a4f-4fbf-bda9-41f837d7d28d` | 196 | Another Prosp sequence |
| `98466961-c5aa-41ac-9b1a-500d5e99e688` | 61 | Another Prosp sequence |

**Fix (D1 relay side):** When the relay replay endpoint enriches the payload with `campaign_id`, also look up the campaign name and add it as `campaign` or `campaign_name` at the top level of the payload. The local extractor will automatically pick it up.

**Fix (local fallback):** If D1 can't provide the name, add a `prosp_campaign_map` table or equivalent to resolve UUID → name locally, and run `reprocess` again.

## Appendix: `reprocess.py` — Implementation Notes for Backend Dev

The reprocess feature:
- Calls `GET /api/v1/replay/events?after_id=X&max_id=Y&platform=Z` on the relay
- Returns raw webhook payloads (same format as normal ingest's `event.get("payload")`)
- Re-runs `extract_relay_fields()`, `extract_campaign_context()`, `extract_bounce_payload()` on each
- Batch-UPDATEs `metadata_json` in-place (no INSERT, no dedup check, no cursor change)
- 1000 events per page, paginated via `next_after_id` cursor

**Current bugs in `reprocess.py`:** Bugs #9 and #10 above (missing import, wrong column name).
