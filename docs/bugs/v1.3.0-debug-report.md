**Date:** 2026-06-30
**Action:** Full DB rebuild (delete + install v1.3.0 + full pull) uncovered multiple data quality issues
**Purpose:** Reference for backend dev to fix data pipeline gaps and implement the `reprocess` feature

---

## Feature Request: `pipeline.py reprocess` — Re-apply current extractors to any ingested data

### The Problem

After a `pull --full`, all 114K events were ingested through old extractor code that missed campaign fields (Prosp/Calendly), bounce diagnostics (all `unknown` type), and sender normalization. The fix was deployed in commit `06ea8e2`, but existing DB rows already have stale extracted data with no way to refresh them.

**There are 4 data kinds, each with different re-processing needs:**

| Kind | Current Cursor | Rows to Re-process | What's Broken |
|---|---|---|---|
| `events` | `last_max_id` | 114,273 | Campaign fields, bounce type, sender normalization |
| `core` (leads) | `last_snapshot_core_after_id` | 151,532 | Lead identity fields, display name parsing |
| `workspace` | `last_snapshot_workspace_after_id` | 60,617 | Stage labels, sentiment, campaign mapping |
| `company` | `last_snapshot_company_after_id` | 41,039 | Company name, domain normalization |

A proper solution must handle all four — otherwise the same problem repeats every time an extractor mapping is fixed.

### Design Constraints

1. **No local payload storage** — PlusVibe emails contain full HTML bodies (50-100K each). Storing raw payloads locally would double or triple the 513MB DB.
2. **No full re-pull** — Re-pulling 367K total records (114K events + 253K snapshots) wastes relay resources and user time.
3. **Targeted** — Must support filtering by kind, ID range, and platform so one-off fixes (e.g. "re-extract only Prosp events from IDs 300K–361K") are fast.
4. **Idempotent** — Running the same range twice must produce the same result (UPDATE, not INSERT).
5. **Minimal relay changes** — The relay already stores everything. New endpoint should be a simple pass-through of stored raw data.

### Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                        pipeline.py reprocess                     │
│                                                                  │
│  1. Determine range & kind from args                             │
│  2. For each kind in scope:                                      │
│     a. Request raw data from relay replay endpoint               │
│     b. For each item:                                            │
│        - Re-run extractors on raw payload                        │
│        - UPDATE existing DB row if anything changed              │
│     c. Print summary of changes                                  │
│  3. Done — no cursors reset, no dedup tables touched             │
└─────────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────────┐
│                        Relay API                                 │
│                                                                  │
│  GET /api/v1/replay                                              │
│    ?kind=events|core|workspace|company                           │
│    &from_id=0                                                    │
│    &to_id=361723                                                 │
│    &platform=prosp,calendly,plusvibe                             │
│    &limit=1000                                                   │
│                                                                  │
│  Returns raw unprocessed payloads in the ID range.               │
│  Paginated (cursor-based).                                       │
│  No business logic applied — just the raw upstream webhook.      │
└─────────────────────────────────────────────────────────────────┘
```

### Relay API Spec

**`GET /api/v1/replay`**

Returns raw stored payloads for any data kind in an ID range.

#### Parameters

| Param | Type | Required | Default | Description |
|---|---|---|---|---|
| `kind` | enum | yes | — | `events`, `core`, `workspace`, `company` |
| `from_id` | int | yes | — | Start ID (inclusive). For events: `last_max_id` range. For snapshots: `last_snapshot_*_after_id` range. |
| `to_id` | int | yes | — | End ID (inclusive) |
| `platform` | string | no | all | Comma-separated: `prosp`, `calendly`, `plusvibe`, `smartlead`, `instantly`, `emailbison` |
| `limit` | int | no | 1000 | Items per page (max 1000) |
| `cursor` | string | no | null | Pagination cursor from previous response |

#### Response

```json
{
  "data": [
    {
      "id": 241520,
      "kind": "events",
      "platform": "prosp",
      "event_type": "send_msg",
      "entity_key": "linkedin.com/in/treybuck",
      "raw_payload": {
        "sender": "linkedin.com/in/treybuck",
        "body": "From linkedin.com/in/treybuck",
        "campaign": "popcam | headshot lounge",
        "campaign_id": 2,
        "webhook_event": "send_msg",
        "relay_received_at": "2026-06-22T16:42:37.855Z"
      },
      "received_at": "2026-06-22T16:42:37.855Z"
    }
  ],
  "next_cursor": "eyJsYXN0X2lkIjogMjQxNTIwfQ==",
  "has_more": true
}
```

For snapshots (`kind=core`), the `raw_payload` is the raw API response from the upstream platform:

```json
{
  "id": 50342,
  "kind": "core",
  "platform": "smartlead",
  "raw_payload": {
    "id": 50342,
    "firstName": "John",
    "lastName": "Doe",
    "email": "john@example.com",
    "linkedinUrl": "...",
    "companyName": "Acme",
    "customFields": { ... },
    "campaignId": 3,
    "campaignName": "popcam | headshot lounge"
  },
  "received_at": "2026-06-15T10:00:00Z"
}
```

**Key detail:** The relay must store the **raw upstream payload as-received** at ingest time — not the processed/normalized version. This is the only way re-processing works when the extractor logic changes.

### Local Implementation: `pipeline.py reprocess`

#### CLI Interface

```
pipeline.py reprocess
  --kind events,core,workspace    # kinds to reprocess (default: events)
  --from 0                         # start relay ID (default: 0)
  --to 361723                      # end relay ID (default: config's last_max_id for the kind)
  --platform prosp,calendly        # platform filter (default: all)
  --dry-run                        # print plan and affected rows, don't execute
  --verbose                        # print per-row diff of changed fields
```

#### Algorithm (pseudocode)

```
function reprocess(args):
    for kind in parse_kinds(args.kind):
        cursor = args.cursor or None
        total_updated = 0
        total_fetched = 0
        total_unchanged = 0

        while True:
            # Fetch raw payloads from relay
            resp = relay_get("/api/v1/replay", {
                "kind": kind,
                "from_id": args.from,
                "to_id": args.to,
                "platform": args.platform,
                "limit": 1000,
                "cursor": cursor
            })

            if args.dry_run:
                total_fetched += len(resp.data)
                cursor = resp.next_cursor
                if not resp.has_more:
                    break
                continue

            for item in resp.data:
                # Find existing row
                if kind == "events":
                    existing = find_event_by_relay_id(item.id)
                elif kind == "core":
                    existing = find_lead_by_relay_id(item.id)
                elif kind == "workspace":
                    existing = find_workspace_lead_by_relay_id(item.id)
                elif kind == "company":
                    existing = find_company_by_relay_id(item.id)

                if not existing:
                    continue  # should not happen during reprocess

                # Re-run extractors on raw payload
                if kind == "events":
                    new_fields = re_extract_event(item.raw_payload)
                    old_fields = extract_from_db_row(existing)
                    diff = compute_diff(old_fields, new_fields)
                    if diff:
                        update_event_row(existing.id, new_fields)
                        total_updated += 1
                        if args.verbose:
                            print_diff(diff)
                    else:
                        total_unchanged += 1

                elif kind in ("core", "workspace", "company"):
                    # Snapshots: re-request from snapshot API
                    # The relay replay endpoint already returns raw snapshot data.
                    # Re-run the snapshot extractor and upsert.
                    new_fields = re_extract_snapshot(item.raw_payload)
                    upsert_lead(existing.id, new_fields)
                    total_updated += 1

            cursor = resp.next_cursor
            if not resp.has_more:
                break

        print_summary(kind, total_fetched, total_updated, total_unchanged)
```

### Why This Is Efficient

| Concern | How It's Addressed |
|---|---|
| **Raw payload size** | Never stored locally — streamed from relay, processed, discarded |
| **114K events** | Only events in the `from_id–to_id` range are fetched |
| **Snapshots** | Same endpoint handles all 4 kinds; each has its own cursor and ID range |
| **Network overhead** | Raw payloads are smaller than processed event objects (no derived fields, no enrichment metadata). ~1KB per event → ~114MB total for a full replay. Streamed in 1000-item pages. |
| **DB load** | Per-row UPDATE only when fields change. No INSERT, no dedup check, no re-verification of idempotency keys. |
| **Platform filter** | Optional — skip platforms that aren't affected by a specific fix |
| **Idempotent** | Running the same range twice: first run updates N rows, second run finds nothing changed → 0 updates |
| **No cursor reset** | `reprocess` doesn't touch config cursors. The pull cursor stays at the latest position. This keeps reprocess completely independent of normal pull operations. |

### Comparison of Approaches

| Approach | Network | Local Storage | Time (est.) | Relay Changes | Future-Proof |
|---|---|---|---|---|---|
| **Re-pull `--full --kind events`** | Full event stream (114K) | None | ~5 min | None | No (snapshots need separate runs) |
| **Store raw in metadata_json** | None | **~200-500MB extra** | Instant | None | Yes, but DB bloat |
| **`reprocess` with relay replay** | Raw payloads only (~114MB) | None | ~1-2 min | New endpoint | Yes (all kinds, filterable) |
| **Re-fetch snapshots individually** | N API calls | None | Slow | None | No (events still broken) |

### Implementation Priority

1. **Relay endpoint**: `GET /api/v1/replay` — stores and returns raw payloads per kind. This is the dependency everything else sits on.
2. **`reprocess` CLI**: Wire up the relay endpoint + local re-extraction loop
3. **Event re-extraction**: Re-run `extract_relay_fields` + `extract_campaign_context` + `extract_bounce_fields` on stored raw payload
4. **Snapshot re-extraction**: Re-run snapshot extractors on raw snapshot data
5. **Dry-run + verbose**: Safety rails before allowing production use

### What This Unlocks

Once built, any future extractor fix follows the same pattern:

```
1. Fix extractor in relay_ingest.py / relay_extractors.py
2. pipeline.py reprocess --kind events --from 300000 --to 361723
3. Done — existing rows updated in seconds
```

No more "these events are broken because they were ingested before the fix."

## Bug #1: `update-manifest.json` includes root-level files — breaks `pipeline.py update`

**Severity:** BLOCKER — prevents `pipeline.py update --tag v1.3.0` from completing

**Root cause:** `update-manifest.json` lists `SECURITY.md` and `references/command-reference.md` in the `files` dict. These are root-level files (live at `skills/outreachmagic/`, not inside `scripts/`). The old `update_download_names()` function only excluded `ROOT_SKILL_FILES` (`README.md`, `install.sh`) and `SKILL.md`.

**Impact:** Any install or update via `--tag v1.3.0` returns 404 on the first root-level file in the manifest.

**Status:** ✅ Fixed in commit `06ea8e2` — switched `update_download_names()` from blacklist to whitelist using `UPDATE_MANIFEST_FILES`.

## Bug #5: All 3,457 bounce events loaded as `unknown` type with no SMTP codes

**Severity:** Medium — bounce tracking is unusable; can't distinguish hard vs soft bounces

**Root cause:** After `pull --full`, all bounce events were loaded into the `bounce_events` table with:
- `bounce_type = 'unknown'` for all 3,457
- `smtp_code = NULL` for all 3,457

The relay either doesn't store bounce type/SMTP code details, or the old extractor didn't map them from the raw payload.

**Raw payload analysis:** The relay webhook payload for bounce events contains fields like `bounce_type`, `type`, `bounce_reason`, `reason`, `error`, `msg`, `body` depending on platform. The new `extract_bounce_fields()` function (added in `06ea8e2`) reads from the raw payload using platform-specific `PLATFORM_BOUNCE_SPECS` in `platform_registry.py`.

**Status:** ✅ Fix deployed in `06ea8e2` for new events. Existing 3,457 are still `unknown` — need `reprocess` to fix them.

## Bug #6: Meeting bookings (Calendly) have no campaign attribution — relay payload doesn't carry it

**Severity:** Medium — can't tell which campaigns drive booked meetings

**Evidence:** All 12 meeting bookings in the last 7 days show `campaign=None`. Raw relay payload for `invitee.created` webhook only contains:

```json
{
  "source": "relay",
  "platform": "calendly",
  "relay_received_at": "...",
  "webhook_event": "invitee.created",
  "sender": "...",
  "relay_id": 360500,
  "body": "...",
  "lead_status_raw": "meeting_booked",
  "lead_status_display": "meeting booked",
  "lead_status_sentiment": "positive"
  // NO campaign field
}
```

**Root cause:** No campaign field exists in the Calendly relay payload. Even if the link was embedded in a campaign email, the booking event fires standalone.

**Status:** ✅ Fix deployed in `06ea8e2` — payload extraction now runs on correct field name. The D1 data migration in `v1.3.1` normalized all stored events to use `payload`. The `event.get("raw")` compat shim was removed. Existing events need `reprocess` to re-apply extractors.

**Permanent fix options:**
1. **Backfill at ingest** — Look up the lead's most recent campaign event before the `meeting_booked` timestamp
2. **Calendly link param** — Add `?campaign=...` to Calendly URLs so the booking callback includes it

## Bug #7: Prosp/LinkedIn webhook payloads don't carry campaign info — 0 of 1,541 events have it

**Severity:** Medium — can't attribute LinkedIn outreach to specific Prosp campaigns/sequences

**Evidence:** Checked all 1,541 Prosp events in the last 30 days — **zero** have a `campaign` field. Raw payload only contains:

```json
{
  "source": "relay",
  "platform": "prosp",
  "relay_received_at": "...",
  "webhook_event": "send_msg",
  "sender": "linkedin.com/in/treybuck",
  "relay_id": 360310,
  "body": "From linkedin.com/in/treybuck"
  // NO campaign field
}
```

Affected event types:

| Event Type | Count (last 30d) | Raw Webhook Event |
|---|---|---|
| linkedin_connect_sent | 426 | `linkedin_connect_sent` |
| linkedin_accept_invite | 209 | `linkedin_accept_invite` |
| linkedin_dm_message_sent | 153 | `linkedin_dm_message_sent` |
| linkedin_message | 134 | `send_msg` / `has_msg_replied` |
| linkedin_connect | 90 | `send_connection` |
| linkedin_dm_reply | 45 | `linkedin_dm_reply` |
| linkedin_connection_accepted | 34 | `accept_invite` |
| withdraw_connection | 18 | `withdraw_connection` |

**Root cause:** Prosp organizes LinkedIn actions into campaigns/sequences, but the webhook payload doesn't carry campaign context. Three possible failure points:
1. Prosp doesn't send campaign/sequence IDs in webhooks
2. Relay receives campaign info but doesn't forward it
3. Extractor ignores campaign field for Prosp events

**Status:** ✅ Fix deployed in `06ea8e2` — payload fallback + campaign extraction in ingest path. Existing events need `reprocess`.

**To debug the relay/Prosp integration:**
1. Check the raw webhook that Prosp sends — does it include `campaign`, `campaign_id`, `sequence_id`, or `campaign_name`?
2. If yes, check if the relay stores it in `raw_payload` but wasn't forwarding it to `event.payload`
3. If no, consider adding `?campaign=CAMPAIGN_NAME` to Prosp webhook URL (Prosp supports custom webhook params)
