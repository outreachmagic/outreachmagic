## Bug #1: `update-manifest.json` includes root-level files — breaks `pipeline.py update`

**Severity:** BLOCKER — prevents `pipeline.py update --tag v1.3.0` from completing

**Root cause:** `update-manifest.json` lists `SECURITY.md` and `references/command-reference.md` in the `files` dict. These are root-level files (live at `skills/outreachmagic/`, not inside `scripts/`). The `update_download_names()` function only excludes `ROOT_SKILL_FILES` (`README.md`, `install.sh`) and `SKILL.md`, so it tries to download from `{scripts_base}/SECURITY.md` → **404 Not Found**.

**Impact:** Anyone running `pipeline.py update --tag v1.3.0` or `pipeline.py update` (which fetches the latest release) gets a 404 error on the first root-level file in the manifest. Also affects `--channel main`.

**Workaround used:** Temporarily removed non-script entries from the manifest in a dev repo checkout, then ran `pipeline.py update` with `dev_repo` config pointing at the local clone.

**Fix suggestion (one of):**
1. Add `SECURITY.md` to `ROOT_SKILL_FILES` in `pipeline.py`
2. Update `update_download_names()` to also filter out root-level files (not just `ROOT_SKILL_FILES ∪ {SKILL.md}`)
3. Remove non-script files (`SECURITY.md`, `references/*.md`, `install.sh`, `README.md`, `SKILL.md`) from the `files` dict in `update-manifest.json`

**Affected code:**
- `ROOT_SKILL_FILES = frozenset({"README.md", "install.sh"})` — line 290, `pipeline.py`
- `update_download_names()` — line 580, `pipeline.py`

## Bug #2: `SECURITY.md` not in `ROOT_SKILL_FILES`

**Severity:** Minor — `SECURITY.md` is silently omitted from the install

**Root cause:** `ROOT_SKILL_FILES` only covers `README.md` and `install.sh`. Even after a successful update, `SECURITY.md` is not copied because the root-file handling loop (`for name in ROOT_SKILL_FILES`) doesn't include it. This is how `SECURITY.md` ended up missing from the install.

**Impact:** The `SECURITY.md` file is absent from a fresh install unless explicitly copied.

**Fix:** Add `"SECURITY.md"` to `ROOT_SKILL_FILES`.

## Bug #3: Massive repeated quarantine warnings per-event (spam-level verbosity)

**Severity:** UX — produces hundreds of KB of repeated output during full pull

**Root cause:** For unmapped campaigns (e.g. `e3 | partnership angle`), every single webhook event generates the same 8-line "To fix this, ask Outreach Magic to..." message. With 4,800+ quarantine events, ~900KB of output was repeated text.

**Impact:** Full-pull output is dominated by duplicated quarantine instructions, making it hard to spot real issues. Terminal scrollback is wasted.

**Fix suggestion:** Deduplicate — print the "To fix this" warning once per campaign/missing-campaign reason, not once per event. A summary line at the end (e.g. "Quarantined 4,855 events from campaign 'e3 | partnership angle' (unmapped)") is sufficient.

## Bug #4: PlusVibe events with no campaign id flood quarantine queue

**Severity:** Medium — 4,800+ items clogging quarantine

**Root cause:** Webhook events from PlusVibe that have no campaign id or name. Each one generates "A webhook event from plusvibe has no campaign id or name" individually.

**Impact:** ~4,855 quarantine items with no actionable resolution path for the user. The queue becomes noise.

## Bug #5: All 3,457 bounce events loaded as `unknown` type with no SMTP codes

**Severity:** Medium — bounce tracking is unusable; can't distinguish hard vs soft bounces

**Root cause:** After `pull --full`, all 3,457 historical bounce events were loaded into the `bounce_events` table with:
- `bounce_type = 'unknown'` for all 3,457
- `smtp_code = NULL` for all 3,457
- No categorization at all (`unknown` is the only type present)

This means the relay either doesn't store bounce type/SMTP code details, or the sync/import path doesn't map them into the schema. The `events` table only shows 3 `email_bounce` events in the last 7 days, so the 3,457 in `bounce_events` are historical — but they're all useless as-is.

**Impact:** Can't run bounce analytics (hard vs soft bounce rates), can't identify problematic SMTP providers or MX servers, can't distinguish permanent rejects from temporary delivery failures.

**Fix suggestions:**
1. Check if the relay API returns bounce type/SMTP code in the webhook payload — if so, map them in the pull/ingest path
2. If the relay doesn't store it, add a fallback enrichment step that classifies bounces by MX/sender domain patterns
3. Verify `bounce_events` schema has correct columns mapped in the extract/ingest pipeline

## Bug #6: Meeting bookings (Calendly) have no campaign attribution — relay payload doesn't carry it

**Severity:** Medium — can't tell which campaigns drive booked meetings

**Evidence:** All 12 meeting bookings in the last 7 days show `campaign=None`. Checked the raw metadata pulled from the relay — the Calendly webhook payload (`invitee.created`) only contains: `source`, `platform`, `relay_received_at`, `webhook_event`, `sender`, `relay_id`, `body`, `lead_status_raw`, `lead_status_display`, `lead_status_sentiment`. **No campaign field exists in the relay payload at all.**

**Root cause:** The relay webhook from Calendly doesn't include campaign context. Even if the Calendly link was embedded in a campaign email, the booking event fires as a standalone action with no reference back to the originating campaign.

**Impact:** Can't measure campaign ROI at the meeting/conversion level. Campaign dashboards underreport conversions.

**Fix suggestions:**
1. Backfill campaign attribution at the ingest layer by looking up the lead's most recent campaign event before the meeting_booked timestamp
2. Or add campaign context in the Calendly webhook integration (pass a `campaign` parameter in the Calendly link)

## Bug #7: Prosp/LinkedIn webhook payloads don't carry campaign info — 0 of 1,541 events have it

**Severity:** Medium — can't attribute LinkedIn outreach to specific Prosp campaigns/sequences

**Evidence:** Checked all 1,541 Prosp events in the last 30 days. **Zero** have a `campaign` field in their metadata. The raw relay payload for Prosp events only contains: `source`, `platform`, `relay_received_at`, `webhook_event`, `sender`, `relay_id`, `body`. No campaign identifier.

This affects all LinkedIn event types:

| Event Type | Count | Raw Webhook Event |
|---|---|---|
| linkedin_connect_sent | 426 | `linkedin_connect_sent` |
| linkedin_accept_invite | 209 | `linkedin_accept_invite` |
| linkedin_dm_message_sent | 153 | `linkedin_dm_message_sent` |
| linkedin_message | 134 | `send_msg` / `has_msg_replied` |
| linkedin_connect | 90 | `send_connection` |
| linkedin_dm_reply | 45 | `linkedin_dm_reply` |
| linkedin_connection_accepted | 34 | `accept_invite` |
| withdraw_connection | 18 | `withdraw_connection` |

**Root cause:** Prosp organizes LinkedIn actions into campaigns/sequences, but the webhook payload sent to the relay doesn't include campaign context. Either:
1. Prosp doesn't send campaign/sequence IDs in its webhooks
2. The relay receives campaign info but doesn't forward it to the ingest layer
3. The ingest/extract layer ignores the campaign field for Prosp events

**Impact:** Can't attribute LinkedIn outreach (connect requests, DMs, replies) to specific Prosp campaigns. Makes it impossible to measure LinkedIn campaign performance.

**Fix suggestions:**
1. Check Prosp's webhook documentation — does it send campaign/sequence IDs in the payload?
2. If yes, verify the relay is forwarding them and the extractor is mapping them
3. If the relay strips them, fix the relay to pass campaign context through
4. If Prosp doesn't send them at all, consider adding campaign to the webhook URL as a query param
