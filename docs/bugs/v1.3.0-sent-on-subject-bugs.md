# Bug Report: Missing Fields in Event Extraction

## Bug #13: `sent_on` (Original Send Timestamp) Not Stored — Relay Received Time Used Instead

**Severity:** HIGH — breaks time-based analytics, makes historical sends appear current

**Symptoms:** All Smartlead and historical events show `created_at` as the relay received time (Sep 26, 2025) rather than the actual email send date. For example:

- Event #116825: actual send was `sent_on: 2024-01-01T14:02:23`, but DB shows `created_at: 2025-09-26T08:23:23`
- The pipeline thinks leads were contacted in Sep 2025 when they were actually contacted in Jan 2024
- Reply rate calculations, follow-up timing, and pipeline stage duration are all wrong

**Root cause:** The Smartlead webhook payload includes a `sent_on` field with the original email send timestamp. The raw payload shape is:

```json
{
  "platform": "smartlead",
  "entity_key": "sabellaschmid@westcliff.edu",
  "event_type": "email_sent",
  "received_at": "2025-09-26T08:23:23.086000",
  "payload": {
    "sent_on": "2024-01-01T14:02:23",
    "campaign_id": "174158",
    "campaign_name": "career services department",
    ...
  }
}
```

The extractor stores `relay_received_at` (when the relay got the webhook) in both `events.created_at` and `metadata_json.relay_received_at`, but never extracts or stores `sent_on` (when the email was actually sent).

**Affected:** All Smartlead events (61,088) and any other platform that sends a `sent_on` timestamp. These represent ~9 months of email sends that appear to have happened on a single day (Sep 26, 2025) instead of their actual dates.

**Impact:**
- Leads that were contacted in Jan-Jun 2024 appear as "contacted Sep 2025"
- Follow-up timing is calculated from the wrong date
- Reply rate analysis is inaccurate
- Pipeline stage durations are compressed
- Cold/warm lead scoring based on recency is unreliable

**Fix options:**
1. **Extract `sent_on` into `metadata_json`** — Add `sent_on` to the Smartlead extractor spec so it's stored as `$.sent_on` in metadata
2. **Use `sent_on` as `created_at`** — When `sent_on` exists, use it as the event timestamp instead of `relay_received_at`. This is the correct behavior — the event timestamp should reflect when the action happened, not when the webhook was received.
3. **Both** — Store `sent_on` plus a `relay_received_at` field for audit

**Affected code:**
- `relay_extractors.py` — Smartlead extractor spec may need `sent_on` field mapping
- `relay_ingest.py` — The `_relay_event_timestamp()` or event creation code that sets `created_at`

## Bug #14: `subject` Field Not Stored for Events

**Severity:** MEDIUM — email subjects are lost, making event identification difficult

**Symptoms:** All events show `subject: NULL` in the `events` table and no `subject` key in `metadata_json`. The `events.subject` column and `metadata_json.subject` are both empty.

**Evidence:**
- Event #4674 (oldest Smartlead send): subject should be "University of Chicago popcam headshots?" but shows `None`
- Event #116825 (replayed from quarantine): subject should be "Sabella, question for Westcliff University" but shows `None`
- All 117,803 events have no subject stored

**Root cause:** The subject is available in the raw payload (Smartlead sends `subject`, PlusVibe sends it through the webhook), but the extractor doesn't map it to `events.subject` or `metadata_json.subject`.

For Smartlead, the raw payload includes:
```json
{
  "payload": {
    "subject": "Sabella, question for Westcliff University",
    ...
  }
}
```

For the Smartlead extractor spec (`_SMARTLEAD_SPEC` or equivalent), `subject` should be mapped in the `event` section:
```python
"subject": ("subject",),
```

**Fix:** Add `subject` to the Smartlead extractor spec's `event` field mappings, and verify the PlusVibe extractor also maps it.

**Affected code:**
- `platform_registry.py` — The Smartlead extractor spec (`_SMARTLEAD_SPEC` or the equivalent for `plusvibe`)
- Needs a row in the `event` mapping like: `"subject": ("subject",)`
- After fix, run `reprocess` to backfill subjects for existing events

4. Check event #116825:
   ```sql

   ```
