# Pull hang postmortem (magic VPS, 2026-06-03)

## Symptoms

- `pull --full` ran ~4.5h, then appeared stuck ~59+ min at `Snapshot (company): p49/~24`
- 0% CPU, no DB mtime change, process alive
- Cursors not persisted on failure (config still 2026-06-02)
- v1.23.7 progress showed `239833/117431` (misleading — see below)

## Root causes

### 1. Client ingest hang (primary)

Progress line is printed **before** `_ingest_relay_page()`. Stall at 0% CPU + frozen DB mtime = blocked or pathologically slow SQLite work on **company** `company_update` rows (5000/page), not relay HTTP.

Contributing factors:

- `ensure_company` / `resolve_company_from_entity_key` use `lower(name) = lower(?)` with **no name index** (domain only).
- `_apply_personalization_payload` opens a connection per company update.
- `get_conn()` had **no `busy_timeout`** (default 5s in Python, but long waits possible under lock).
- Cursors saved **only at end of full `sync_from_relay_org()`** — killed runs lose all progress.

### 2. v1.23.7 progress UI bug (secondary)

Before v1.23.8, snapshot progress used global `snap_total` and core’s `pending_snapshot_count` for all kinds. **Do not use those ratios for relay diagnostics.**

### 3. Relay (ruled out for hang)

`pullSnapshotsByKind` is `ORDER BY id LIMIT n`; `has_more_snapshots = rows.length >= limit`. No infinite server loop observed.

## VPS recovery (2026-06-03)

1. Machine rebooted (~22 min uptime at check).
2. Skill at **1.23.9**.
3. Advanced **core** / **workspace** cursors from `relay_ingested` max relay ids:
   - `last_snapshot_core_after_id`: 6293 → **122641**
   - `last_snapshot_workspace_after_id`: 5879 → **120620**
   - Company: almost no `relay:3*` marks (page 49 never finished ingest) — left at **259**
4. Patched `pipeline.py` / `db_conn.py` on VPS (checkpoint cursors, snapshot stall guard, busy_timeout).
5. Incremental `pull` in tmux: `~/hermes/pull-live.log`

### 4. `migrate_db()` re-runs `backfill_companies_from_leads()` every pull

On a 116k-lead DB this rescans all leads and calls `ensure_company` on every pull invocation before relay sync starts — looks like a hang (99% CPU, no log output).

**Fix:** only backfill when `companies` is empty.

### 5. Company `IntegrityError: FOREIGN KEY constraint failed` (285 rows)

`ingest_agent_entry` called `ensure_company()` (INSERT) then `conn.close()` **without `commit()`**, so the new `companies` row was rolled back. `apply_agent_company_sync_payload()` then inserted `company_personalization` with a stale `company_id`.

**Fix:** `conn.commit()` before close on the `company_update` path. Repair replay imported **+285 companies** (43,746 → 44,031).

**Note:** `company_update` returns `lead_id=None`, so pull stats show `filtered` not `imported` even on success — misleading only in logs.

## Ingest performance (why ~80s / 5k company rows, ~4 min / 5k events)

Per-row costs dominate:

| Cost | Company snapshots | Webhook events |
|------|-------------------|----------------|
| `get_conn()` + `commit()` per row | 2 connections/row (resolve + apply) before fix | Multiple per event |
| `relay_already_ingested()` SELECT | Once per row even after pull prefetch | Same |
| `get_org_routing_config()` | Loaded for every agent row (unnecessary for `company_update`) | Routing + quarantine path |
| SQLite | `lower(name)` lookup without index (mitigated: `idx_companies_name_lower`) | Lead resolve + event insert |

**Repo optimizations (post-incident):** single-connection `company_update`, skip dedupe SELECT when `defer_mark`, dedicated company path without routing load.

**Further wins:** one transaction per pull page (`executemany` marks + batch ingest), event cursor checkpoint per page, optional WAL `synchronous=NORMAL` for bulk pull.

## v1.24.0 pull throughput (client-side)

- One SQLite connection per event pull (not per page): bulk PRAGMAs, campaign routing cache, batched dedupe + workspace idempotency prefetch
- One commit per page: ingest no longer commits per row when `pull_conn` is shared; relay marks + activity refresh batched at page end
- `resolve_lead` / `update_lead_stage` reuse pull connection during ingest
